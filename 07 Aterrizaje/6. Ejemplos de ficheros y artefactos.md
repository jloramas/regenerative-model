# Ejemplos de ficheros y artefactos

Este documento presenta **ejemplos concretos de ficheros, estructuras y contenidos** utilizados en el modelo descrito en los documentos anteriores.

⚠️ **Advertencia importante** Los ejemplos que siguen:

* no son definitivos,  
* pueden contener inconsistencias,  
* no pretenden cubrir todos los casos,  
* ni deben tomarse como una especificación cerrada.

Su objetivo es **servir como material de arranque**, facilitar las primeras pruebas reales y permitir que los formatos finales **emerjan de la práctica**, no del diseño teórico.

---

## 1\. Declaración central de artefactos

Ejemplo de fichero de declaración explícita y centralizada:

```
# artifacts.yaml

artifacts:
  - id: ui.empty-state
    path: app/ui/empty-state
    cycle: exploratorio
    kind: ui

  - id: page.patients
    path: app/pages/patients
    cycle: exploratorio
    kind: page

  - id: layout.master-detail
    path: catalog/layouts/master-detail
    cycle: exploratorio
    kind: layout
```

Observaciones:

* El esquema es deliberadamente simple.  
* Los campos no son obligatorios ni definitivos.  
* El `id` es la identidad estable del artefacto.  
* El `path` define su frontera física.

---

## 2\. Artefacto simple (un solo fichero)

Ejemplo de artefacto UI muy simple:

```
app/ui/empty-state.tsx
```

Contenido orientativo:

```
export function EmptyState() {
  return (
    <div>
      <h2>No data available</h2>
      <p>Please add your first item.</p>
    </div>
  )
}
```

Notas:

* No hay metadata inline obligatoria.  
* La intención vive fuera (en la declaración y/o specs).  
* Es válido mientras la complejidad sea baja.

---

## 3\. Artefacto compuesto (directorio)

Ejemplo de artefacto más completo:

```
app/pages/patients/
  index.tsx
  spec.md
  verify.md
```

### spec.md (propuesta)

```
# Intent

Página principal de gestión de pacientes.

## Objetivo
Permitir visualizar, buscar y acceder al detalle de pacientes.

## Fuera de alcance
- Edición avanzada
- Integraciones externas
```

---

### verify.md (propuesta)

```
# Verification

- La lista de pacientes es visible al cargar la página.
- Es posible acceder al detalle de un paciente.
- La página no bloquea la navegación principal.
```

Estos ficheros:

* no son tests clásicos,  
* son contratos verificables,  
* y pueden ser usados por workflows IA.

---

## 4\. Especificaciones transversales

Ejemplo de requisitos que afectan a múltiples artefactos:

```
specs/
  ux.md
  ai.md
```

### ux.md (fragmento)

```
- Evitar modales bloqueantes en flujos principales.
- Priorizar feedback inmediato en acciones del usuario.
```

Estas specs:

* no son artefactos por defecto,  
* pero condicionan generación y verificación.

---

## 5\. Reglas IA del proyecto (propuesta)

Ejemplo muy simplificado:

```
ai/
  rules.md
```

```
# IA Rules

## General
- Respetar artefactos declarados.
- No regenerar sin intención explícita.

## Artefacts
- Mantener coherencia intención ↔ realización ↔ verificación.
- No eliminar verificación sin autorización.
```

Este fichero:

* gobierna workflows,  
* no describe implementación,  
* puede evolucionar rápidamente.

---

## 6\. Ejemplo de workflow IA: generar artefacto

Ejemplo conceptual (no técnico):

**Input**

* artefacto: `page.patients`  
* intención actualizada  
* ciclo: exploratorio

**Pasos**

1. Leer declaración del artefacto.  
2. Cargar specs locales y transversales.  
3. Generar o actualizar realización.  
4. Proponer ajustes de verificación.  
5. Señalar incoherencias si existen.

**Output**

* cambios en `index.tsx`  
* cambios en `verify.md`  
* warnings si procede

---

## 7\. Ejemplo de workflow IA: verificación

**Input**

* artefacto: `page.patients`

**Pasos**

1. Leer intención.  
2. Analizar realización.  
3. Contrastar con criterios de verificación.  
4. Emitir resultado semántico (ok / warning / mismatch).

No se asume:

* ejecución real,  
* ni tests automáticos, solo **consistencia semántica**.

---

## 8\. Ejemplo de trabajo fuera de artefactos

Ejemplo de script técnico:

```
scripts/seed-db.ts
```

Aquí:

* se aplican reglas técnicas del proyecto,  
* no se exige intención ni verificación semántica,  
* la IA puede modificarlo con mayor libertad, pero respetando estándares del proyecto.

---

## 9\. Qué validar con estos ejemplos

Estos ejemplos sirven para comprobar:

* si la declaración central es suficiente,  
* si los artefactos son operables,  
* si la IA puede actuar con contexto limitado,  
* si la verificación es útil sin ser pesada.

Las inconsistencias que aparezcan **son valiosas**: indican dónde el modelo necesita ajuste.

---

## 10\. Cierre

Estos ejemplos no son el destino, sino el punto de partida.

El objetivo ahora es:

* usarlos,  
* romperlos,  
* ajustarlos,  
* y dejar que los formatos reales **emerjan del trabajo diario**.
